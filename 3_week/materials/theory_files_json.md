**Файлы**

Открытие всегда через `with open()` — закрывает файл автоматически даже при ошибке.

Режимы: `'r'` читать, `'w'` писать (затирает), `'a'` дописывать в конец, `'rb'`/`'wb'` бинарный режим.

Лучший способ читать — `for line in f` — поток, не грузит всё в память.

Пути всегда через `os.path.join()`, не склеивать строками.

**JSON**

JSON — это строка. Python-объект — это объект. Это разные вещи.

`dumps` / `loads` — работа со строкой в памяти. `dump` / `load` — работа с файлом. `s` = string.

```python
json.dumps(obj)   # dict → str
json.loads(str)   # str → dict
json.dump(obj, f) # dict → файл
json.load(f)      # файл → dict
```

JSON умеет только: dict, list, str, int, float, bool, None. Datetime, set, numpy — не умеет, нужна конвертация.

Из API всегда приходит JSON-строка. `response.json()` в requests делает `json.loads` автоматически.

**CSV**

Каждая строка — список. Первая строка обычно заголовки.

Для аналитических задач CSV сразу читай через pandas `pd.read_csv()` — не через модуль `csv`. Модуль `csv` только если pandas избыточен.

**Сериализация объектов**

JSON — универсально, читаемо, для всего простого.

`joblib.dump` / `joblib.load` — только для sklearn моделей.

`np.save` / `np.load` — только для numpy массивов и эмбеддингов.

Pickle — не использовать.

**Потоки**

Поток = последовательность байтов + позиция в ней. `fp` = указатель на источник.

Важно когда: файл большой (читай построчно), данные приходят по частям (HTTP streaming, LLM токены), пишешь логи непрерывно (режим `'a'`).

Сводка по памяти:

Да, останется. `with` закрывает файловый поток, но объект `loaded_response` живёт в памяти до конца скрипта или пока не выйдет из области видимости.

**Как Python управляет памятью**

Python использует подсчёт ссылок (reference counting). У каждого объекта есть счётчик — сколько переменных на него ссылается. Как только счётчик падает до 0 — объект удаляется автоматически.

```python
loaded_response = json.load(fp)  # счётчик = 1
loaded_response = None           # счётчик = 0 → память освобождается
# или
del loaded_response              # то же самое
```

**В твоём коде конкретно**

`loaded_response` — это список словарей, весь JSON в памяти. Для биржевого фида на тысячи тикеров — не проблема, килобайты. Для 500 PDF по экономике в RAG — уже думаешь об этом.

**Когда это будет важно в твоих задачах**

Эмбеддинги. Когда будешь считать векторы для документов — каждый эмбеддинг это numpy массив float32. 1000 документов × 1536 размерность (OpenAI) = ~6MB только эмбеддингов в памяти. Умножь на батчи — начинает иметь значение.

```python
for batch in chunks(documents, size=100):
    embeddings = model.encode(batch)
    qdrant.upsert(embeddings)
    del embeddings  # явно освобождаем после записи в Qdrant
```

Большие датафреймы pandas. Загрузил CSV хроматограммы, сделал анализ, результат сохранил — исходный DataFrame больше не нужен.

```python
df_raw = pd.read_csv('chromatogram.csv')
df_clean = process(df_raw)
del df_raw  # освобождаем ~2x память
```

**Garbage collector**

Помимо счётчика ссылок есть GC для циклических ссылок (A ссылается на B, B на A — счётчик никогда не падает до 0). Для твоих задач это экзотика, не думай об этом.

**Практическое правило:** не думай об этом пока скрипт не падает с `MemoryError`. Когда упадёт — `del` на большие объекты после использования. В 90% аналитических задач память не проблема.
