1. Что такое MCP

MCP (Model Context Protocol) — открытый стандарт, разработанный Anthropic, для подключения AI-приложений к внешним системам. Опубликован в 2024 году, сейчас поддерживается Claude, ChatGPT, VSCode Copilot, Cursor и другими.

Точная формулировка: MCP — это протокол для обмена контекстом между LLM-приложением (хостом) и внешними инструментами/данными (серверами). Он стандартизирует как инструменты описываются, вызываются и как возвращают результат.

До MCP каждый AI-клиент делал свою интеграцию по-своему: Claude — через function calling в API, GPT — через plugins, Cursor — через своё API. Каждый инструмент нужно было писать заново под каждого клиента. MCP решает это как USB решил проблему разных разъёмов.

2. Аналогия USB-C

USB-C аналогия:

До USB-C у каждого устройства был свой разъём. Хочешь подключить принтер к ноуту — нужен специфический провод для этой пары.

MCP — это USB-C для AI. Один стандартный протокол. Написал MCP-сервер один раз → он работает с Claude, GPT, Cursor, любым хостом который поддерживает MCP.

Конкретно это значит:

  • Playwright MCP написан один раз → работает и в Claude Desktop, и в Claude Code, и в Cursor
  • Твой кастомный MCP-сервер для fin_advisor → подключается к любому AI без переписывания
  • Сообщество пишет серверы (GitHub, Slack, PostgreSQL, Notion) → ты просто подключаешь

3. Архитектура: хост, клиент, сервер

Три участника:

MCP Host — AI-приложение (Claude Code, Claude Desktop, Cursor)
↕ создаёт и управляет
MCP Client — компонент внутри хоста, одно соединение = один клиент
↕ JSON-RPC через stdio или HTTP
MCP Server — внешняя программа (Playwright, Serena, твой сервер)

Хост (Claude Code) — главный. Он запускает серверы, создаёт клиентов для каждого из них, передаёт список доступных инструментов в LLM.

Клиент — объект внутри хоста. Один клиент = одно соединение с одним сервером. Если подключено 3 сервера — 3 клиента.

Сервер — отдельная программа (или процесс). Может быть локальной (запускается как дочерний процесс) или удалённой (работает где-то в сети). Она не знает ничего о LLM — только отвечает на запросы по протоколу.

4. Как работает протокол (JSON-RPC)

MCP использует JSON-RPC 2.0 — простой протокол вызова удалённых процедур через JSON. Каждое сообщение — это JSON-объект.

Жизненный цикл соединения

1
Initialize — клиент отправляет initialize с версией протокола и своими capabilities. Сервер отвечает своими capabilities (какие примитивы поддерживает).
2
Discovery — клиент запрашивает tools/list. Сервер возвращает список инструментов с описаниями и JSON Schema параметров. Хост добавляет их в контекст LLM.
3
Execution — LLM решает вызвать инструмент. Хост отправляет tools/call с аргументами. Сервер выполняет действие, возвращает результат. Хост добавляет результат в разговор.
4
Notifications — сервер может сам уведомить клиента об изменениях (например, список инструментов обновился). Клиент не опрашивает — просто получает уведомление.

Пример: вызов инструмента

// Хост → Сервер: вызвать инструмент
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "browser_navigate",
    "arguments": { "url": "https://www.ft.com/" }
  }
}

// Сервер → Хост: результат
{
  "jsonrpc": "2.0",
  "id": 3,
  "result": {
    "content": [{ "type": "text", "text": "Navigated to https://www.ft.com/ — page loaded" }]
  }
}

5. Примитивы: Tools, Resources, Prompts

Примитив Что это Пример
Tools Функции которые агент может вызвать для действия browser_click, query_db, send_email
Resources Данные/контекст которые агент может прочитать Содержимое файла, схема БД, результат API
Prompts Шаблоны системных промптов от сервера "Ты аналитик. Вот данные: {data}"

Для тебя сейчас главное — Tools. Агент видит список инструментов как список функций с описаниями. Когда LLM понимает что нужно кликнуть по кнопке — она генерирует вызов browser_click с нужными параметрами, хост отправляет это в сервер, сервер кликает, результат возвращается.

6. Транспорт: stdio vs HTTP

Тип Как работает Когда использовать
stdio Хост запускает сервер как дочерний процесс, общение через stdin/stdout. Сервер живёт пока живёт хост. Локальные серверы (Playwright, Serena, файловая система). Самый простой вариант.
HTTP/SSE Сервер работает отдельно, хост подключается по HTTP. Поддерживает много клиентов одновременно. Удалённые серверы, корпоративные интеграции (Sentry, Notion, GitHub).

Playwright MCP и Serena — stdio. Claude Code запускает их как процессы, общается через stdin/stdout. Ты не видишь эти процессы явно — Claude Code управляет ими сам.

7. MCP Playwright — что даёт агенту

50+ инструментов для управления браузером. Агент больше не пишет код с Playwright — он просто вызывает инструменты как команды.

Основные инструменты

ИнструментЧто делает
browser_navigateПерейти на URL
browser_clickКликнуть по элементу (по тексту/роли)
browser_typeВвести текст в поле
browser_snapshotПолучить accessibility tree страницы (текстовое представление DOM)
browser_take_screenshotСкриншот для визуальной проверки
browser_wait_forЖдать появления элемента

Snapshot режим — ключевое отличие от обычного Playwright

Вместо скриншотов сервер отдаёт accessibility tree — структурированное текстовое представление страницы. Агент получает что-то вроде:

button "Accept cookies" [id=sp_choice_type_11]
link "Markets" href="/markets"
heading "Global Economy" level=1

Это в 10–50x токенов дешевле чем описывать скриншот. И детерминированно — агент точно знает что кликнуть.

Как агент находит кнопку consent banner

Агент вызывает browser_snapshot, видит в дереве button "Accept" [class=sp_choice_type_11], затем вызывает browser_click с этим селектором. Никакого hardcode — агент сам разберётся по дереву.

8. MCP Serena — что даёт агенту

Serena — инструменты для работы с кодовой базой на уровне символов, не строк. Это как дать агенту функциональность IDE.

ИнструментЧто делает
find_symbolНайти функцию/класс по имени во всём проекте
find_referencing_symbolsНайти все места где используется символ
insert_after_symbolВставить код сразу после определения функции
get_symbol_definitionПолучить полное определение символа
search_symbols_in_fileСписок всех символов в файле
Зачем это нужно? Claude Code по умолчанию читает файлы целиком. Для большого проекта (fin_advisor с десятками модулей) это дорого по токенам и медленно. Serena позволяет агенту работать точечно — "найди функцию X, посмотри где она вызывается, вставь изменение". Это важно когда проект вырастет.

9. Установка — пошаговая инструкция

Шаг 1: Установить Node.js (нужен для Playwright MCP)

Зачем Node.js?

Playwright MCP — это JavaScript-пакет (@playwright/mcp), опубликованный в npm (реестр пакетов Node.js). Команда npx @playwright/mcp@latest — это "скачай и запусти npm-пакет".

Node.js — среда выполнения JavaScript вне браузера. Как Python — среда для .py файлов, так Node.js — среда для .js/.ts файлов.

npm — менеджер пакетов Node.js (как pip для Python).

npx — утилита из npm: скачать пакет + сразу запустить, не устанавливая глобально. Аналог uvx в Python-экосистеме.

Без Node.js команда npx не существует → MCP-сервер не запустится → Claude Code не сможет подключиться к Playwright. 
curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt-get install -y nodejs
node --version   # должно быть v22.x
npx --version    # должно появиться автоматически вместе с node

Шаг 2: Установить uv и uvx (нужен для Serena)

Зачем uv и uvx?

uv — это современный менеджер Python-пакетов и окружений, написанный на Rust. Быстрее pip в 10–100x. Умеет создавать venv, устанавливать пакеты, управлять версиями Python.

uvx — субкоманда uv: запустить Python-инструмент в изолированном окружении без установки. Это точный аналог npx для Python.

Serena — Python-пакет. Команда uvx --from git+https://... serena start-mcp-server значит:
"скачай serena прямо с GitHub, создай временное venv, запусти как MCP-сервер".

Почему не просто pip install serena? Потому что Serena не опубликована в PyPI — только на GitHub. И uvx изолирует зависимости Serena от твоего проекта.

10. Как агент использует MCP в твоём проекте

Сценарий: агент сам чинит сломанный consent-клик

Ты говоришь агенту: "Кнопка consent banner не нажимается, проверь и почини"

Агент (с Playwright MCP):

  1. Вызывает browser_navigate → заходит на ft.com
  2. Вызывает browser_snapshot → смотрит дерево страницы
  3. Находит кнопку, видит реальный класс/текст
  4. Вызывает browser_click → проверяет что сработало
  5. Читает код ft.py (через Serena или Read)
  6. Исправляет селектор в refresh_cookies_playwright()

Без MCP vs с MCP

Без MCPС MCP Playwright
Найти нужный элемент Ты идёшь в браузер, копируешь селектор, пишешь агенту Агент сам заходит, смотрит дерево, находит
Протухли куки Ты открываешь DevTools, копируешь куки, обновляешь .env Агент сам логинится, забирает куки, обновляет сессию
Сайт изменил вёрстку Код ломается, ты ищешь новый селектор Агент получает ошибку, сам смотрит новую структуру, чинит

Схема для ft.py: агент + MCP

FTClient.refresh_cookies_playwright()
  → если ошибка / кнопка не найдена
  → агент получает exception
  → вызывает browser_navigate(ft.com)
  → вызывает browser_snapshot()
  → анализирует дерево
  → находит новый селектор
  → редактирует ft.py
  → повторяет попытку

Это и есть агент-петля с инструментами. Фаза 1 твоего плана (апрель–май) — именно это: LangGraph/PydanticAI агент + MCP-инструменты + петля с обработкой ошибок.

Короткий итог:

MCP = стандартный разъём для AI-инструментов.
Хост (Claude Code) → создаёт клиент → соединяется с сервером → LLM видит инструменты → вызывает их через JSON-RPC → результат идёт обратно в контекст.

Playwright MCP даёт агенту браузер. Serena даёт агенту понимание кода.
Вместе — агент может сам разбираться с внештатными ситуациями без тебя.

11. Как это реально работает — механизм в деталях

Твоё понимание: верно ли?

Да, понял правильно. Уточню механизм точнее:

  1. Хост (Claude Code) — это ты как пользователь + LLM + оболочка. Он знает о всех подключённых MCP-серверах.
  2. При старте хост создаёт MCP-клиент для каждого сервера и запрашивает tools/list — список доступных инструментов. Этот список добавляется в системный промпт LLM.
  3. LLM видит что-то вроде: "У тебя есть инструмент browser_click(selector) — кликнуть по элементу в браузере".
  4. Когда в коде вылетает RuntimeError("consent_banner_selector_outdated") — LLM получает это в контекст и сама решает вызвать browser_navigate, потом browser_snapshot.
  5. Хост перехватывает это намерение LLM, отправляет JSON-RPC запрос tools/call на MCP-сервер Playwright.
  6. Сервер Playwright (локальный процесс, запущенный через npx) открывает браузер, выполняет действие, возвращает результат.
  7. Результат (accessibility tree страницы) возвращается в контекст LLM. LLM видит структуру, находит кнопку, генерирует правку в ft.py.

Важное уточнение: "агентик-клиент" в твоей формулировке — это не отдельная сущность, это сама LLM внутри хоста, которая принимает решения. MCP-клиент — просто объект-соединение, не агент. Агент = LLM + хост + инструменты вместе.

// Упрощённо: что происходит когда вылетает ошибка в ft.py

[ft.py]          RuntimeError: consent_banner_selector_outdated
      ↓
[Claude Code]    LLM получает ошибку в контексте
                 LLM решает: "надо посмотреть страницу через браузер"
                 LLM генерирует вызов инструмента browser_navigate
      ↓
[MCP Client]     отправляет JSON-RPC → { "method": "tools/call",
                                          "params": { "name": "browser_navigate",
                                                      "arguments": { "url": "https://ft.com" } } }
      ↓
[MCP Server]     (npx @playwright/mcp — локальный процесс)
                 открывает Chromium, переходит на ft.com
                 возвращает: { "result": { "content": [{ "type": "text", "text": "Navigated" }] } }
      ↓
[Claude Code]    LLM вызывает browser_snapshot()
                 получает accessibility tree: "button 'Accept' [class=sp_choice_type_11]"
                 видит актуальный селектор
                 редактирует ft.py → готово

12. Среды выполнения: Python, Node.js — как это работает

Что такое "среда" (runtime)

Процессор понимает только машинный код — последовательность байтов, каждый из которых означает конкретную инструкцию (сложить, записать в память, перейти по адресу). Высокоуровневые языки — Python, JavaScript — человекочитаемый текст. Процессор не понимает for item in items:.

Среда выполнения — это программа-посредник, которая берёт твой код и делает из него то, что процессор понимает. Два основных подхода:

ПодходКак работаетПример
Компиляция Весь код сразу переводится в машинный код → получается .exe. Среда нужна только при компиляции. C, Rust, Go
Интерпретация Код читается и выполняется строка за строкой в реальном времени. Среда нужна при каждом запуске. Python, JavaScript
Гибрид (JIT) Интерпретируется, но горячие участки кода компилируются на лету для скорости. Java, Node.js V8, PyPy

Как работает Python

Шаги от .py файла до результата:

  1. Лексер — разбивает текст на токены: for, item, in, items, :
  2. Парсер — строит AST (Abstract Syntax Tree) — дерево структуры кода
  3. Компилятор — переводит AST в байткод (.pyc файлы в __pycache__/) — промежуточный низкоуровневый формат
  4. CPython VM — виртуальная машина, выполняет байткод инструкцию за инструкцией

Байткод — не машинный код. Это инструкции для виртуальной машины Python. Поэтому один .py файл запускается на Windows, Linux, Mac без перекомпиляции — везде своя CPython VM, но байткод одинаковый. 
# Посмотреть байткод Python — это реально существует:
import dis
dis.dis("for item in items: print(item)")

# Вывод:
#   1     LOAD_NAME    0 (items)
#         GET_ITER
#   >>    FOR_ITER     4 (to 14)
#         STORE_NAME   1 (item)
#         LOAD_NAME    2 (print)
#         LOAD_NAME    1 (item)
#         CALL         1
#         POP_TOP
#         JUMP_BACKWARD 6 (to 4)

Как работает JavaScript / Node.js

JavaScript изначально создавался только для браузеров — каждый браузер встраивал свой JS-движок. Google для Chrome написал V8 — очень быстрый движок с JIT-компиляцией.

Node.js — это V8, вытащенный из браузера и упакованный как самостоятельная программа. Плюс добавили API для работы с файлами, сетью, процессами — то, чего в браузере нет по соображениям безопасности.

Ключевое отличие Node.js от Python:

Node.js изначально построен на event loop + async I/O. Одним потоком обслуживает тысячи соединений одновременно — не блокируется на ожидании ответа от сети или диска. Именно поэтому Node.js стал стандартом для серверов и инструментов реального времени.

Python добавил async/await позже (3.4+), и он менее органичен. В Node.js async — это DNA, не надстройка.

npx и uvx — точный механизм

Твоё понимание верно: npx/uvx — это "воспользоваться инструментом, не загружая его полностью глобально".

Но точнее: они всё равно скачивают пакет — просто во временную или кэш-папку, а не в глобальную среду. Разница:

npm install -g @playwright/mcp — устанавливает глобально в /usr/lib/node_modules/. Остаётся навсегда. Версия одна на всю систему.

npx @playwright/mcp@latest — скачивает в кэш (~/.npm/_npx/), запускает, кэш можно удалить. Всегда берёт последнюю версию если не указана конкретная. Не загрязняет глобальную среду.

Аналогично uvx: скачивает в ~/.cache/uv/, создаёт изолированный venv, запускает, не трогает твой проект.

Почему Playwright MCP написан на Node.js, а не Python

  • Playwright — библиотека Microsoft, изначально написана на TypeScript/Node.js. Python-версия — это биндинги поверх Node.js ядра.
  • Node.js async I/O идеально подходит для MCP-сервера: он ждёт команды от хоста, выполняет действие в браузере, возвращает результат — всё асинхронно, без блокировок.
  • MCP-протокол разработан Anthropic в экосистеме TypeScript → эталонные реализации на Node.js. Python SDK появился позже.
Итог по средам:

Язык → текст который ты пишешь.
Среда (runtime) → программа которая этот текст выполняет (CPython для Python, V8/Node.js для JS).
Пакетный менеджер → инструмент чтобы скачать чужой код (pip, npm).
npx/uvx → скачать и запустить без постоянной установки. Пакет всё равно скачивается — просто в кэш, не в глобальную среду.

MCP-серверы могут быть написаны на любом языке — важно только что они говорят по протоколу JSON-RPC. Playwright MCP — Node.js. Serena — Python. Твой будущий сервер — на чём удобно.
