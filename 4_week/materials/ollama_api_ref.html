Ollama API — Справочник

HTTP, порты, аутентификация, совместимость с Anthropic SDK, ответы моделей

Блок 0 — Сервер, хост, порт
localhost / 127.0.0.1

Одно и то же. localhost — человекочитаемое имя, резолвится в 127.0.0.1 через /etc/hosts. Это loopback-интерфейс — виртуальный, пакет не покидает ОС, физической сети нет.

Порт 11434

IP доставляет до машины — порт доставляет до процесса. ОС маршрутизирует входящий пакет нужному процессу по номеру порта. 11434 — произвольный выбор команды Ollama, просто конвенция.

0.0.0.0 — опасно

Слушает все интерфейсы — и loopback, и WiFi-адаптер. Кто угодно в той же сети может достучаться. HTTP не шифрует — тело запроса, заголовки, ключи читаемы через Wireshark.

127.0.0.1 — безопасно локально

Пакет идёт через loopback, физически не выходит на WiFi-карту. Перехватывать нечего. HTTP без HTTPS — норм. Как только выставляешь на VPS или 0.0.0.0 — нужен nginx + TLS.

bash — проверить на каком адресе слушает Ollama
# требует sudo — процесс запущен от другого пользователя
sudo lsof -i :11434

# хороший вывод: только localhost
ollama  2138 ollama  TCP localhost:11434 (LISTEN)

# плохой вывод: торчит наружу
ollama  2138 ollama  TCP 0.0.0.0:11434 (LISTEN)

# пинг сервиса
curl http://localhost:11434
# → Ollama is running
Блок 1 — Почему HTTP норм локально
Я подключён к WiFi — разве HTTP не небезопасен?
Нет, если слушаешь 127.0.0.1. Loopback и WiFi — разные сетевые интерфейсы в ОС. Трафик на 127.0.0.1 никогда не попадает на WiFi-адаптер физически. HTTPS нужен когда пакет идёт через чужую сеть. Здесь сети нет вообще — только память ОС.
Блок 2 — Аутентификация
Ollama локально

Аутентификация не требуется. Никаких заголовков Authorization. Сервис доступен только с локальной машины — это и есть защита.

Anthropic SDK

API-ключ передаётся при создании клиента → SDK автоматически добавляет его в заголовок x-api-key каждого запроса. Хранишь в .env, достаёшь через python-dotenv.

Блок 3 — Форматы ответов: Ollama vs Anthropic
Могу ли я обратиться к сохранённому ответу по id через Anthropic API?
Нет. id в ответе Anthropic — это идентификатор конкретного запроса для логирования и отладки, не ключ хранилища. Anthropic не хранит твои сообщения для последующего доступа. Модель stateless — каждый запрос независим. История = то, что ты сам передаёшь в messages.
Поле Ollama /api/generate Anthropic /v1/messages
текст ответа response.json()["response"] response.content[0].text
токены входа prompt_eval_count usage.input_tokens
токены выхода eval_count usage.output_tokens
причина остановки done_reason: "stop" stop_reason: "end_turn"
время инференса eval_duration (наносек.) нет
id запроса нет id: "msg_..." (только для логов)
Блок 4 — Anthropic SDK → Ollama (совместимость)
Могу ли я использовать Anthropic SDK для запросов к Ollama?
Да. Ollama реализует эндпоинт /v1/messages совместимый с Anthropic API. Меняешь только base_url и api_key (ключ любой — он игнорируется). Весь остальной код — тот же. messages, tools, stream, system — работают одинаково.
python — один клиент, два бэкенда
import anthropic
from dotenv import load_dotenv
import os

load_dotenv()

# ── Anthropic (облако) ──────────────────────────────
client_cloud = anthropic.Anthropic(
    api_key=os.getenv("ANTHROPIC_API_KEY")
)

# ── Ollama (локально) ───────────────────────────────
client_local = anthropic.Anthropic(
    base_url="http://localhost:11434",
    api_key="ollama"   # обязательно, но игнорируется
)

# ── Запрос — одинаковый код для обоих ───────────────
def ask(client, model, text):
    msg = client.messages.create(
        model=model,
        max_tokens=1024,
        messages=[{"role": "user", "content": text}]
    )
    return msg.content[0].text

# облако
ask(client_cloud, "claude-sonnet-4-5", "Привет")

# локально
ask(client_local, "gemma3:4b", "Привет")
Когда всё же стоит использовать requests напрямую, а не SDK?
Три случая: (1) нужен эндпоинт которого нет в SDK — например /api/tags (список моделей), /api/pull (скачать модель), /api/show — это нативные Ollama-эндпоинты, Anthropic SDK их не знает. (2) минимальные зависимости — скрипт на сервере где не хочешь тянуть SDK. (3) отладка и изучение — видишь сырой HTTP, понимаешь что происходит. Для основной работы с моделями — SDK предпочтительнее.
Блок 5 — Контекст и память
Ollama помнит историю сама?
Нет. Модель stateless. /set history в интерактивном режиме — это история команд терминала (стрелка вверх), не контекст разговора. История = ты сам передаёшь весь список messages в каждом запросе.
python — накопление истории вручную
messages = []

def chat(user_input):
    messages.append({"role": "user", "content": user_input})
    
    response = client.messages.create(
        model="gemma3:4b",
        max_tokens=1024,
        messages=messages   # вся история каждый раз
    )
    
    answer = response.content[0].text
    messages.append({"role": "assistant", "content": answer})
    return answer
Параметр Назначение Пример
num_ctx Контекстное окно — суммарный лимит токенов (все сообщения + ответы) 10000 → ~18 обменов по 550 токенов
num_predict Макс. токенов в одном ответе модели 1024
temperature Случайность ответа. 0 = детерминировано, 1+ = творчески 0.7
Важно: когда окно переполняется — Ollama обрезает старые сообщения. Твой Python-код всё ещё хранит полный список. Можешь сам управлять: всегда оставлять system-промпт + последние N обменов.
Блок 6 — Эндпоинты Ollama
Эндпоинт CLI аналог Назначение
POST /api/generate ollama run Запрос к модели, одиночный промпт
POST /api/chat ollama run Мультитёрн с messages[]
GET /api/tags ollama list Список скачанных моделей
POST /api/pull ollama pull Скачать модель
POST /api/show ollama show Инфо о модели
POST /api/delete ollama rm Удалить модель
POST /v1/messages — Anthropic-совместимый эндпоинт (SDK работает)
ollama serve — единственная команда которую нельзя заменить API-вызовом. Сервер должен быть запущен до любых запросов. У тебя запускается автоматически через systemd при старте системы.
