## WHEN TO USE THIS SKILL (MANDATORY CONDITIONS)

Применять когда пользователь:
- Говорит "сделай практику по [тема/файл]"
- Говорит "подготовь задания для [неделя/тема]"
- Говорит "хочу попрактиковаться с [инструмент/концепт]"
- Ссылается на конкретный theory-файл или guide из текущей недели

## WHEN THIS SKILL MUST NOT BE USED

- Пользователь просит объяснить теорию без задач — только отвечать текстом
- Пользователь просит проверить/исправить уже написанный код — это не practice
- Тема не определена и пользователь не указал откуда брать контекст

## PURPOSE

Этот скилл:
- Авторизует чтение файлов текущей недели (`N_week/`) и плана `AI_Engineer_Plan.html`
- Авторизует чтение `.fin_advisor/` для понимания какие операции/паттерны полезны
- Создаёт файлы практики в `N_week/` — только `.py` файлы
- Запрещает раскрывать детали замысла fin_advisor в создаваемых файлах
- Запрещает копировать код из `.fin_advisor/`

## CORE RULES (NON-NEGOTIABLE)

1. **Читать перед созданием** — всегда читать theory-файл недели перед генерацией задач
2. **Финансовый домен** — все примеры и задачи используют: тикеры, котировки, новостные источники, сигналы, ликвидность, RSS-фиды, API конфиги. Никакой химии, никаких абстрактных foo/bar
3. **Публичность** — файлы идут в PUBLIC репо. Ничего из fin_advisor замысла напрямую: без упоминания конкретной системы, без имён реальных источников из `.fin_advisor` если они несут контекст
4. **Формат файла** — строго: блок ТЕОРИЯ (комментарий), блок ПРИМЕР (рабочий код), блок ЗАДАЧА (комментарий + заготовка). Тот же стиль что в `2_week/*.py`
5. **Максимум 2 файла за раз** — `01_*.py`, `02_*.py`. Не больше
6. **Вспомогательные данные** — если нужен input-файл (JSON, CSV) для задачи, создавать в `N_week/data/`
7. **Уровень задачи** — не слишком глубоко, мышечная память + читаемость кода. Пользователь будет делать это сам руками, не через агента
8. **Без финального резюме** — не объяснять после создания что сделал, файлы говорят сами за себя

## PROTOCOL (STRICT ORDER)

### Step 1 — Определить контекст
- Прочитать theory-файл(ы) текущей недели на которые ссылается пользователь
- Если не указана неделя — спросить
- Прочитать `.fin_advisor/FinAdvisor_Organism.html` (скрипты-органеллы, блок источников) чтобы понять какие операции актуальны

### Step 2 — Определить тему и операции
- Из theory-файла: что изучается (инструменты, паттерны, концепты)
- Из fin_advisor: какие скрипты/операции на этой же технологии — брать как вдохновение для домена задачи, не копировать

### Step 3 — Спланировать файлы
- Решить: 1 или 2 файла (1 — если тема узкая, 2 — если есть две отдельные концепции)
- Определить нужны ли data-файлы

### Step 4 — Создать файлы
- Создать data-файлы если нужны
- Создать `01_*.py` и при необходимости `02_*.py`
- Придерживаться формата: ТЕОРИЯ → ПРИМЕР → ЗАДАЧА

### Step 5 — Подтверждение
Вывести только: список созданных файлов с кратким описанием (одна строка каждый).

## BAD EXAMPLES

### Bad 1 — Раскрытие замысла
```python
# Задача: реализуй парсер для FinAdvisor системы,
# которая отслеживает ликвидность через FT и Bloomberg RSS
```
Почему плохо: называет систему и конкретные источники из fin_advisor — публичное репо.

### Bad 2 — Абстрактный домен
```python
# Задача: распарси список фруктов из HTML
```
Почему плохо: нарушает правило финансового домена.

### Bad 3 — Слишком сложно
```python
# Задача: реализуй async Playwright скрейпер с retry,
# rate limiting, прокси-ротацией и сохранением в Qdrant
```
Почему плохо: избыточная сложность для мышечной памяти. Пользователь делает руками.
